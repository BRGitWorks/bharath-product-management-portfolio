Day 1 — Foundations

Speak the model’s language using C-R-E-A-M: Context, Role, Examples, Ask, Meta constraints (tone, format, length).

Starter template 
You are a <role> helping with <domain>.
Context: <concise background and audience>.
Task: <what you must produce>.
Constraints: bullet list of must-haves and must-nots.
Output format: <JSON/markdown/table>.
Voice: <tone>.
Good 
You are a senior technical writer.
Context: Create a quickstart for a Windows desktop app.
Task: Draft a 6-step install guide for beginners.
Constraints: 8th-grade reading level, numbered steps, bold UI labels.
Output: Markdown with an H2 title and a tip box at the end.
Voice: clear, friendly.
Bad 
Write installation steps.
Try it yourself

Rewrite one of your recent prompts using C-R-E-A-M. Aim to cut ~30% words while adding explicit constraints. Paste your original and improved versions below your notes.

Day 4 — Decomposition & Planning

Get better answers by breaking work into steps and planning before writing.

Plan → Draft → Check loop 
First, list a 3-step plan for the task.
Then produce the draft.
Finally, return a checklist of 5 verifications you ran.
Ask for uncertainties 
Before answering, list the top 3 assumptions that could be wrong.
If any are critical, ask clarifying questions; otherwise proceed.
Constrained thinking 
Reason step by step only to yourself and output the final result.
(If steps are needed externally, present a concise bullet plan.)

Day 5 — Tools, Schemas & Structured Output

Get reliably structured outputs the first time.

JSON schema guardrail 
Return ONLY valid JSON matching this schema:
{
  "title":"string",
  "audience":"string",
  "steps":[{"n":"integer","text":"string"}],
  "risks":["string"],
  "next_steps":["string"]
}
If unknown, use null. No extra keys.
Tables 
Make a Markdown table with columns: Step | User action | System response | Edge cases.
Exactly 6 rows.
Fill N/A if unsure.
Function-call intent (pseudo) 
If you need data, propose a function call with name and arguments.
Return JSON: {"call":{"name":"fetch_user","args":{"id":"123"}}}

Day 6 — Evaluation, Tests & Safety

Create repeatable checks so quality doesn’t drift.

Mini test harness 
Given inputs: [<case1>, <case2>, <case3>]
Produce outputs and a pass/fail table against these rules:
- Must include an explicit instruction
- Must specify output format
- Must include constraints (length/tone/audience)
Red-team prompts 
Stress-test the prompt for: ambiguity, missing context, unsafe asks, proprietary data leakage.
Return: a bullet list of issues and fixes.

Hallucination minimizers

  • Ask for sources or mark unverifiable claims.
  • Constrain to supplied text (“answer only from…”).
  • Prefer extractive answers when facts matter.

Day 7 — Workflows & Automation

Turn good prompts into reusable systems.

Prompt to Playbook 
Title: <use-case>
Inputs: <required fields>
System prompt: <role+rules>
User prompt template: <template with {{placeholders}}>
Output shape: <JSON/table/text>
Quality checks: <list>
Multi-turn pattern 
Round 1: Collect requirements (goal, audience, data, format).
Round 2: Propose outline/options.
Round 3: Deliver draft.
Round 4: Review against rubric → final.

Versioning & notes

  • Name prompts semantically: csense-install-v3
  • Keep a change log and test set.
  • Pin constraints; vary examples when needed.

Cheat-sheet

Command verbs

  • Draft / Rewrite / Summarize / Translate
  • Classify / Extract / Compare / Critique
  • Plan / Outline / Decompose / Evaluate

Constraints

  • Audience & reading level
  • Length (words, bullets, tokens)
  • Style (tone, voice, persona)
  • Format (JSON keys, table columns)

Quality rubric

  1. Clear goal stated in one sentence.
  2. Enough context to avoid guessing.
  3. Explicit constraints (format, tone, length).
  4. Tested with at least three varied inputs.
  5. Maintained (versioned with notes).

Common prompt fixes

  • Too vague → Add task + format + constraints.
  • Waffly tone → Specify voice and audience.
  • Unstructured output → Demand JSON/table.
  • Model makes things up → Restrict to provided text or ask for sources.
  • Inconsistent results → Provide few-shot examples.
